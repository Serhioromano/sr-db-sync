# Обязательные требования

## 1. Обновление документации после работы с TypeScript

После завершения работы с любым файлом TypeScript (`.ts`, `.tsx`) необходимо:
- Обновить `README.md`
- Обновить `.pi/SYSTEM.md`
- Обновить `CHANGELOG.md`

## 2. Проверка через Fallow

После изменения любого файла TypeScript необходимо проверить его при помощи утилиты **Fallow**.

### Установка и использование Fallow

Fallow — утилита для анализа dead code, дубликатов и здоровья кодовой базы.

```bash
# Установка (уже добавлена как devDependency)
npm install

# Быстрый запуск (все проверки разом)
npx fallow

# Отдельные проверки
npx fallow dead-code     # неиспользуемый код
npx fallow dupes         # дубликаты
npx fallow health        # сложность и кандидаты на рефакторинг

# Превью авто-исправлений
npx fallow fix --dry-run

# JSON-вывод (для агентов/CI)
npx fallow --format json
```

114 встроенных плагинов покрывают Next.js, Vite, Ember, Jest, Tailwind и другие фреймворки. Первый запуск не требует конфигурации.

Ссылка на документацию: https://fallow.tools/docs/quickstart/

## 3. Покрытие тестами после работы с TypeScript

После завершения работы с любым файлом TypeScript (`.ts`, `.tsx`) необходимо:
- Написать тесты, покрывающие внедрённую функциональность
- Сразу же исполнить тесты и убедиться, что они проходят

### Тестовая инфраструктура — OneTest (Bun)

В качестве тестового фреймворка используется **OneTest** — встроенный рантайм Bun (`bun test`). Никаких дополнительных зависимостей не требуется.

- Все тесты находятся в директории `test/`
- Именование: `test/*.test.ts`
- Для тестов используется локальная база **SQLite** — не требует сетевых подключений
- Файлы SQLite создаются прямо в `test/` (временные базы — `:memory:` или файловые `test/*.sqlite`)
- Тестовая структура базы данных описана в `test/test.dbml` — этот файл служит эталонной схемой для проверки парсинга DBML, генерации DBML, работы snash и migrate
- Файлы `.sqlite` добавлены в `.gitignore` (не коммитятся)
- API тестов: `describe`, `it`, `expect`, `beforeAll`, `afterAll`, `beforeEach`, `afterEach` — импорт из `bun:test`
- Моки: `mock.module()`, `spyOn()` — встроены в Bun

### Запуск тестов

```bash
bun test                  # все тесты в test/
bun test --watch          # автоперезапуск при изменениях
bun test --coverage       # отчёт о покрытии
```
